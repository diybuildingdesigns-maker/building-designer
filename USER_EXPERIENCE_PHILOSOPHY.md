# DIY Building Designs Professional
# USER_EXPERIENCE_PHILOSOPHY.md

## Purpose

This document defines how users should experience DIY Building Designs Professional.

The goal is not simply to create CAD software. The goal is to create software that allows people to design steel buildings without requiring them to think like structural engineers.

---

# Core Philosophy

Users design the building.

The software designs the structure.

The engineering engine validates and optimizes the structural system behind the scenes while allowing qualified users to review or modify it when necessary.

---

# Three User Modes

## 1. Designer Mode

Designed for:

- Sales staff
- Designers
- Drafters
- New users

The interface should focus on:

- Building geometry
- Doors and windows
- Colors
- Roof style
- Visual editing
- Customer presentation

Engineering decisions remain automatic.

---

## 2. Professional Mode

Designed for experienced steel building designers.

Adds access to:

- Member overrides
- Secondary framing controls
- Material options
- Bay customization
- Advanced framing tools

Professional Mode builds on Designer Mode without replacing it.

---

## 3. Engineering Mode

Designed for licensed engineers and advanced users.

Adds access to:

- Design loads
- Code settings
- Deflection limits
- Connection settings
- Structural calculations
- Design reports

Engineering Mode should expose the engineering engine without changing the core workflow.

---

# Progressive Disclosure

The interface should reveal complexity only when it is needed.

New users should never be overwhelmed by engineering settings.

Experienced users should never be slowed down by beginner tutorials.

---

# Optional Design Assistant

The software may include a Design Assistant.

The assistant should:

- Guide first-time users.
- Explain unfamiliar concepts.
- Recommend next steps.
- Warn about common mistakes.

The assistant must be optional.

Users must be able to disable:

- Popups
- Walkthroughs
- Hints
- Tutorials
- Suggestions

Once disabled, the assistant should remain out of the way.

---

# Confidence, Not Confusion

The software should communicate status in plain language.

Examples:

- Design Ready
- Needs Review
- Structural Issue Detected

Whenever practical, explain why a recommendation changed without requiring engineering knowledge.

---

# Every Action Has One Goal

Every feature should help users:

- Design faster
- Make fewer mistakes
- Produce better drawings
- Generate accurate material lists
- Deliver professional projects

---

# Long-Term Vision

DIY Building Designs Professional should feel less like traditional CAD software and more like an intelligent design partner.

Users should leave the program feeling that they designed the building while the software handled the repetitive engineering and documentation work.
